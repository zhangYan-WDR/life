# 先落核心业务，识别入口先占位

## Summary

- 二期先实现可交付的核心链路：菜谱库、随机选菜、点餐审批、点餐历史、缺料分析。

- 图片识别相关能力这期

  不真正接入外部云 API

  ，只做入口占位与手动兜底：

  - 点击“小票识别入冰箱”或“图片识别存菜谱”时，提示“功能正在规划中，请先使用手动添加”。
  - 同时提供直达手动流程的跳转，避免入口点了无反馈。

- 后端为 OCR/多模态识别预留清晰的 TODO 扩展位，但本期不开放正式识别接口，也不引入图片存储。

## 实施阶段

### 第一阶段：核心业务先上线

- 新增菜谱模型：
  - recipes：菜名、基准份数、步骤、备注、状态、创建人。
  - recipe_ingredients：食材来源、名称快照、分类快照、数量、单位、排序。
- 新增点餐模型：
  - meal_requests
  - meal_request_recipes
  - meal_request_responses
- 新增核心能力：
  - 菜谱列表、创建、编辑、详情。
  - 一次点多道菜，按目标份数相对基准份数缩放。
  - 全员审批：任一人拒绝即拒绝，全部同意才通过。
  - 点餐记录留存，支持 待我处理 / 我发起的 / 全部历史。
  - 点餐详情内查看汇总食材与缺料分析。
- 随机选菜：
  - 从当前家庭所有启用菜谱中随机抽 1 道。
  - 支持“换一个”和“点这个菜”。
  - “点这个菜”直接进入发起点餐页并自动带入该菜。

### 第二阶段：识别入口先占位

- 首页或工作台新增两个入口：
  - 小票识别入冰箱
  - 图片识别存菜谱
- 本期点击行为固定为：
  - 先弹出提示：功能正在规划中，先用手动添加吧
  - 提供明确跳转：
    - 小票入口跳到冰箱手动新增页。
    - 图片菜谱入口跳到菜谱手动创建页。
- 前端不要出现“即将支持识别后自动导入”这类半成品表单，避免用户误以为可以上传成功。

### 第三阶段：后端预留 OCR TODO

- 在后端新增识别服务抽象，但仅留空实现和注释，不接真实供应商：
  - ImageRecognitionService
  - analyzeReceipt(...)
  - analyzeRecipeImage(...)
- 返回策略：
  - 本期不暴露正式 controller 路由。
  - 若必须先建接口壳，则统一返回业务错误：功能正在规划中，请先手动处理。
- 配置层预留：
  - app.ocr.provider
  - app.ocr.api-key
  - app.ocr.enabled
- README/开发说明中明确标注：
  - OCR 为 TODO
  - 未来由后端代理外部云 API
  - 小程序不直连第三方识别服务

## Key Changes

- 首页工作台扩展为 4 类能力：
  - 冰箱库存
  - 家庭食材
  - 菜谱库
  - 点餐单
- 首页新增辅助入口：
  - 今晚吃什么
  - 小票识别入冰箱（规划中）
  - 图片识别存菜谱（规划中）
- 缺料分析规则维持不变：
  - 按名称聚合。
  - 支持重量类、容量类常见换算。
  - 包装类只同单位比较。
- 系统食材目录本期一起扩充，尽量覆盖市面常见食材，减少家庭手工维护。

## Public APIs / Interfaces

- 本期正式开放：
  - GET /recipes
  - GET /recipes/{id}
  - POST /recipes
  - PUT /recipes/{id}
  - GET /recipes/random
  - POST /meal-requests
  - GET /meal-requests?view=PENDING|MINE|ALL
  - GET /meal-requests/{id}
  - POST /meal-requests/{id}/respond
  - GET /meal-requests/{id}/ingredient-gap
  - POST /subscriptions/meal-request-reminder
- 本期不正式开放 OCR 接口。
- 若需要保留占位接口，统一使用明确的未开放响应，不做假上传成功。

## Test Plan

- 菜谱：
  - 能手动创建、编辑、查看详情。
  - 可从系统食材和家庭食材中选择。
- 随机选菜：
  - 能随机推荐。
  - 能继续换菜。
  - 能直接带入点餐流程。
- 点餐：
  - 支持多道菜。
  - 支持按份数缩放。
  - 支持全员审批与历史留存。
- 缺料分析：
  - 充足、不足、缺失三种结果正确。
  - 重量/容量换算正确。
- 规划中入口：
  - 点击后提示文案正确。
  - 能顺利跳到手动新增页。
  - 不出现上传后卡住或空白页面。

## Assumptions

- 本期目标是先把“能真正用起来的家庭做饭协作链路”上线，不把工期压在 OCR 接入上。
- OCR/图片存储/封面图全部延后到下一阶段；这期不引入对象存储、不保留上传图片。
- “功能正在规划中”入口不是纯占位按钮，而是带手动兜底跳转的可用入口。
